Blast Motion Putting Pipeline

A full-stack data platform for golf putting analytics. An automated Python pipeline scrapes putting sensor data from the Blast Connect API, loads it into PostgreSQL, and exposes it through a FastAPI backend consumed by a Next.js frontend. Airflow orchestrates the pipeline on a 2-hour schedule.

Architecture

Pipeline Runner

The pipeline authenticates via CSRF token, fetches new sessions using a high watermark, cleans the data, and upserts into PostgreSQL — all within a single atomic transaction.

main.pyPython
def run_pipeline():
    logger.info("Starting Blast Motion pipeline")
    try:
        session = requests.Session()

        # Step 1-2: Authenticate
        token = get_csrf_token(session)
        login(session, token)

        # Step 3: Fetch all sessions
        sessions_list, sessions_data = get_sessions(session)

        # Step 4: Incremental load via high watermark
        engine = get_engine()
        max_session_id = int(get_high_watermark(engine))
        logger.info(f"High watermark: {max_session_id}")

        # Step 5: Only process new data
        new_sessions = [s for s in sessions_list if int(s['id']) > max_session_id]

        if not new_sessions:
            logger.info("No new sessions — exiting")
        else:
            # Step 6-7: Extract & transform
            all_putts = get_putts(session, new_sessions)
            df_putts = clean_putts_df(pd.DataFrame(all_putts))
            sessions_df = clean_sessions_df(pd.DataFrame(new_sessions))

            # Step 8: Atomic load
            with engine.begin() as conn:
                upsert_sessions(sessions_df, engine)
                upsert_putts(df_putts, engine)
                logger.info("Transaction committed")
    except BlastAuthError as e:
        logger.error(f"Auth failed: {e}")
        raise

Airflow DAG

Scheduled every 2 hours with automatic retry. A single PythonOperator wraps the pipeline runner.

        python_callable=run_pipeline,

Incremental Loading

The pipeline uses a high watermark query to avoid reprocessing data. Only sessions with IDs greater than the current max are fetched from the API.

get_high_watermark.sqlSQL
SELECT
    COALESCE(max(blast_session_id), 0)
FROM src.blast_sessions;

Idempotent Upserts

Uses PostgreSQL's ON CONFLICT DO NOTHING for safe, repeatable writes. If the pipeline runs twice, no duplicates are created.

utils.pyPython
def upsert_sessions(df: pd.DataFrame, engine) -> None:
    records = df.to_dict(orient='records')
    with engine.begin() as conn:
        stmt = insert(get_sessions_table(engine)).values(records)
        stmt = stmt.on_conflict_do_nothing(
            index_elements=['blast_session_id']
        )
        conn.execute(stmt)
        logger.info(f"Upserted {len(records)} sessions")

FastAPI Backend

The analysis platform exposes a REST API with JWT authentication, CRUD endpoints for rounds and putts, and real-time strokes gained calculation.

services/shots_gained.pyPython
def calculate_shots_gained(
    db: Session,
    distance_ft: float,
    made: bool,
    distance_remaining_ft: float | None,
    skill_level: str = "tour",
) -> float | None:
    """SG = Expected_Putts(start) - Expected_Putts(end) - 1"""
    expected_start = get_expected_putts(db, distance_ft, skill_level)
    if expected_start is None:
        return None

    if made:
        return round(expected_start - 1.0, 3)

    if distance_remaining_ft is None:
        return None

    expected_end = get_expected_putts(db, distance_remaining_ft, skill_level)
    return round(expected_start - expected_end - 1.0, 3)
The shots gained dimension table includes benchmarks across 5 skill levels (tour, scratch, low/mid/high handicap), allowing users to compare their putting against any tier.

Resilient HTTP Client

The pipeline includes exponential backoff with jitter for transient failures, and fails fast on auth or rate-limit errors.

utils.pyPython
def get_with_retry(session, url, params=None, retries=MAX_RETRIES, base_wait=BASE_WAIT):
    for attempt in range(retries):
        try:
            response = session.get(url, params=params, timeout=TIMEOUT)
            if response.status_code == 200:
                return response
            elif response.status_code == 401:
                raise BlastAuthError("Unauthorized")
            elif response.status_code == 429:
                raise BlastRateLimitError("Rate limited")
            elif response.status_code >= 500:
                raise BlastServerError(f"Server error {response.status_code}")
        except (ConnectionError, Timeout, BlastServerError) as e:
            if attempt < retries - 1:
                # Exponential backoff + jitter
                wait = (base_wait ** attempt) + random.uniform(0, 1)
                time.sleep(wait)
            continue
    raise BlastAPIError(f"All {retries} attempts failed")

Database Schema

Two-schema design: src for raw sensor data ingested by the pipeline, raw for user-entered context (rounds, practice sessions, putts with shots gained).

    blast_session_id    INT           PRIMARY KEY,
    session_type_id     INT           NOT NULL,
    session_name        VARCHAR(255)  NOT NULL,
