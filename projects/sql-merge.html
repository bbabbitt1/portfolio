SQL MERGE: Staging-to-Production Pipeline

A two-script SQL pipeline for loading Replicon timesheet data from a staging table into production. The first script cleanses malformed activity codes, then both scripts use the MERGE statement to perform atomic upserts — updating existing records and inserting new ones in a single pass.

The Scenario

Replicon timesheet data lands in a staging table via an API integration. The data needs to be cleansed (malformed activity codes), type-cast (date fields), and merged into the production table. Records that already exist should be updated; new records should be inserted. The MERGE statement handles both in a single atomic operation, keyed on TimeEntryId.

Script 1 — Cleanse & Load

This script handles dirty data before merging. Activity codes sourced from the API sometimes contain full activity names instead of short codes.

1 Fix malformed ActivityCode values

Valid activity codes are 4–5 characters. When the code is outside that range, extract the first word from ActivityName as a fallback. The CHARINDEX trick appends a space so single-word names don't break.

-- Fix activity codes that are too short or too long UPDATE staged_replicon_data SET ActivityCode = LEFT(ActivityName, CHARINDEX(' ', ActivityName + ' ') - 1) WHERE LEN(ActivityCode) NOT BETWEEN 4 AND 5;
Why not CASE? A targeted UPDATE...WHERE only touches rows with bad codes. A CASE expression inside the MERGE would evaluate on every row — even the ones with valid codes — adding unnecessary computation to the merge operation.
2 MERGE — match on TimeEntryId

The MERGE statement joins staging to production on the natural key. WHEN MATCHED updates existing records; WHEN NOT MATCHED BY TARGET inserts new ones. This is a single atomic operation — no risk of partial loads.

MERGE dbo.replicon_timesheet_prod AS target USING dbo.staged_replicon_data AS source ON target.TimeEntryId = source.TimeEntryId
3 WHEN MATCHED — update existing records

When a TimeEntryId already exists in production, overwrite every attribute with the latest staged values. LastUpdatedOn = GETDATE() stamps when the record was last refreshed — useful for debugging and freshness checks downstream.

WHEN MATCHED THEN UPDATE SET TimeEntryType = source.TimeEntryType, UserId = source.UserId, UserName = source.UserName, EntryDate = source.EntryDate, DurationHours = source.DurationHours, WorkTimeDurationHours = source.WorkTimeDurationHours, BreakTimeDurationHours = source.BreakTimeDurationHours, TimeOffTimeDurationHours = source.TimeOffTimeDurationHours, ActivityId = source.ActivityId, ActivityName = source.ActivityName, ActivityCode = source.ActivityCode, TimesheetId = source.TimesheetId, TimesheetStartDate = source.TimesheetStartDate, TimesheetEndDate = source.TimesheetEndDate, -- ... remaining columns LastUpdatedOn = GETDATE()
4 WHEN NOT MATCHED — insert new records

New time entries that don't exist in production are inserted with all columns plus a LastUpdatedOn timestamp. The BY TARGET qualifier makes the intent explicit — only rows missing from the target are inserted.

WHEN NOT MATCHED BY TARGET THEN INSERT ( TimeEntryId, TimeEntryType, UserId, UserName, EntryDate, DurationHours, WorkTimeDurationHours, BreakTimeDurationHours, TimeOffTimeDurationHours, ActivityId, ActivityName, ActivityCode, TimesheetId, TimesheetStartDate, TimesheetEndDate, -- ... remaining columns LastUpdatedOn ) VALUES ( source.TimeEntryId, source.TimeEntryType, source.UserId, source.UserName, source.EntryDate, source.DurationHours, source.WorkTimeDurationHours, source.BreakTimeDurationHours, source.TimeOffTimeDurationHours, source.ActivityId, source.ActivityName, source.ActivityCode, source.TimesheetId, source.TimesheetStartDate, source.TimesheetEndDate, -- ... remaining columns GETDATE() );

Script 2 — Type-Safe MERGE

A second version of the merge that adds defensive type casting for date fields sourced from the API as strings.

5 TRY_CAST for safe date conversion

The Replicon API sometimes returns date fields as strings. TRY_CAST safely converts them — returning NULL instead of failing the entire merge on a bad value. Applied to EntryDate, TimesheetStartDate, and TimesheetEndDate in both the UPDATE and INSERT clauses.

MERGE dbo.replicon_timesheet_prod AS target USING dbo.stg_replicon_data AS source ON target.TimeEntryId = source.TimeEntryId WHEN MATCHED THEN UPDATE SET -- Date fields use TRY_CAST for safe conversion target.EntryDate = TRY_CAST(source.EntryDate AS DATE), target.TimesheetStartDate = TRY_CAST(source.TimesheetStartDate AS DATE), target.TimesheetEndDate = TRY_CAST(source.TimesheetEndDate AS DATE), -- Non-date fields pass through directly target.UserName = source.UserName, target.DurationHours = source.DurationHours, target.ActivityCode = source.ActivityCode, -- ... target.LastUpdatedOn = GETDATE() WHEN NOT MATCHED THEN INSERT (TimeEntryId, EntryDate, TimesheetStartDate, ...) VALUES ( source.TimeEntryId, TRY_CAST(source.EntryDate AS DATE), TRY_CAST(source.TimesheetStartDate AS DATE), ... );
TRY_CAST vs CAST: CAST will throw an error and abort the entire MERGE if a single date value is malformed. TRY_CAST returns NULL for bad values, letting the merge complete. You can then identify and fix the nulls downstream.

Key Patterns

✓ Cleanse before merge

Fix data quality issues in the staging table before the MERGE runs. This keeps the MERGE statement focused on its job (upsert logic) and makes cleansing rules independently testable.

✓ Atomic upsert with MERGE

A single MERGE replaces a multi-step pattern of IF EXISTS → UPDATE ELSE INSERT. It's atomic — if any part fails, the whole operation rolls back. No partial loads, no orphaned inserts.

✓ Defensive typing at the boundary

Use TRY_CAST at the system boundary (API → database) where data types aren't guaranteed. Internal queries between typed tables don't need this overhead.

✓ Audit column (LastUpdatedOn)

GETDATE() on every upsert provides a built-in freshness indicator. Downstream consumers can filter to recent loads, and stale records surface immediately in monitoring queries.
